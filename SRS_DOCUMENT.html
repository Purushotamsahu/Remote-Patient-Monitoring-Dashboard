1. Introduction

1.1 Purpose

This SRS establishes the formal requirements framework for the MediFlow platform. It acts as the core baseline for systems architects, frontend developers (React), backend engineers (Laravel), and QA testing teams during implementation.

1.2 Scope

MediFlow is a Remote Patient Monitoring application utilizing a secure cloud-native model to track, alert, and summarize physiological telemetry.

Primary Objectives:

  • Dynamic Custom Dashboards: Dedicated Patient (Care Hub), Doctor (Command Center), and Admin workspaces.
  • Biometric Acquisition: Tracking metrics (Heart Rate, SpO2, Blood Pressure, Sugar Levels, Temperature).
  • Immediate Warning Pushes: Threshold violations broadcasted to doctors under 500ms via WebSockets.
  • AI Diagnostics Support: Summary draft creation utilizing Google Gemini 2.5 Flash API.
  • Vetted Scheduling: Conflicting calendar slots prevented during doctor bookings.

Out of Scope: Legally binding primary prescriptions and direct hardware clinical intervention commands.

2. General Description

2.1 Product Perspective

MediFlow functions as a modular cloud application. The frontend React single-page application dispatches REST queries to a stateless Laravel 12 API, utilizing MongoDB Atlas for database storage. Asynchronous warning triggers are processed via persistent WebSockets over Laravel Reverb servers.

The system connects directly to three external actors (Patients, Doctors, and Administrators) and external APIs (Google OAuth 2.0 and Google Gemini 2.5 Flash).

2.2 Product Functions

  • Role Authorization: Portal views restricted by parsed token roles.
  • Biometric logs: Stores SpO2, Heart Rate, Temperature, BP, and Sugar Levels.
  • Real-time warnings: WebSocket dispatch of metrics outside safe parameters.
  • Clinical Calendars: Time slot booking and conflict verification.
  • AI Diagnostics: Generates report insights based on historical vitals.
  • Doctor Vetting: Administration review panel for medical certificates.

2.3 User Characteristics

  • Patients: Require simple visual numeric logs, large status badges, and simplified booking calendars.
  • Doctors: Require complex vital charts, active alert list tick panels, and rapid AI summary buttons.
  • Administrators: Require secure vetting logs, system activity trackers, and user state controls.

2.4 General Constraints

  • Compliance: Enforces secure HTTPS data in transit and encrypted MongoDB databases (HIPAA).
  • Stateless session: User access mapped solely via JWT header packets.
  • Application verification: Database referential constraints checked in Laravel code layers.

2.5 Assumptions and Dependencies

  • Network: Constant internet required to reach MongoDB Atlas and Google API servers.
  • External availability: Depends on Google Gemini 2.5 Flash API quotas and OAuth operational states.

3.2 Functional Requirements

3.2.1 User Authentication & Authorization (RBAC)

3.2.1.1 Introduction

Restricts access to designated patient, doctor, and admin portals based on role access keys.

3.2.1.2 Inputs

  • Local authentication email and password credentials.
  • Social token parameters through Google OAuth logins.

3.2.1.3 Processing

  1. Checks for password format validations and duplicate email matches.
  2. Applies bcrypt password encryption with a standard work index of 12.
  3. Constructs signed JWT token packets containing user ID and authorization roles (valid for 60m).

3.2.1.4 Outputs

  • Secure JSON responses containing role profiles and signed JWT token strings.
  • Dynamic client routing redirects based on authorization status.

3.2.1.5 Error Handling

  • Returns a 422 error response for invalid formats and duplicate registration attempts.
  • Returns a 401 error response for invalid login matching.

3.2.2 Real-time Health Vital Monitoring & Tracking

3.2.2.1 Introduction

Handles the entry, database write, visual graphing, and alert checking of biometrics.

3.2.2.2 Inputs

  • Numerical vitals: Heart Rate, SpO2, Temperature, BP, and Sugar Levels.
  • Metadata: Patient ID string and entry timestamps.

3.2.2.3 Processing

  1. Verifies metric inputs lie within standard physiological bounds.
  2. Applies MongoDB writes to the health_metrics collection linked to the patient ID.
  3. Fires asynchronous alert checking algorithms and triggers Reverb broadcasts if limits are exceeded.

3.2.2.4 Outputs

  • Updated history charts on the patient home care hub.
  • Updated visual vital curves on doctor workspaces.

3.2.2.5 Error Handling

  • Dispatches 422 error validations if metrics exceed predefined maximum parameters.

3.2.3 Appointment Scheduling & Booking System

3.2.3.1 Introduction

Schedules clinical sessions between patients and certified doctors.

3.2.3.2 Inputs

  • Selected Doctor ID, target date, consultation hour slot, and clinical notes.
  • Workflow actions: approve, reject, or request reschedule.

3.2.3.3 Processing

  1. Checks that target physician profiles possess verified licensing states.
  2. Queries database collection tables to prevent doctor calendar double-bookings.
  3. Writes scheduled state documents and dispatches WebSocket notification flags.

3.2.3.4 Outputs

  • Updated calendar slots in both the patient and doctor dashboards.
  • A new document written in the appointments collection.

3.2.3.5 Error Handling

  • Returns a 409 conflict code if booking overlaps with an existing appointment.

3.2.5 Real-Time Critical Alerts & Notifications

3.2.5.1 Introduction

Warns assigned doctors instantly when a patient enters critical vital states.

3.2.5.2 Inputs

  • Biometric inputs: SpO2, Heart Rate, Sugar levels, or BP readings.
  • Defined warning bounds: SpO2 < 94%, Heart Rate > 120 bpm.

3.2.5.3 Processing

  1. Compares incoming telemetry with predefined vital threshold variables.
  2. Writes critical alert log documents to the database when thresholds are crossed.
  3. Broadcasts the AlertCreated event via Laravel Reverb to active doctor channels.

3.2.5.4 Outputs

  • Visual emergency popups and audio alarms on doctor dashboards.

3.2.5.5 Error Handling

  • If WebSockets disconnect, the frontend dispatches regular API calls every 15 seconds to ensure continuity.

3.5 Non-Functional Requirements

3.5.1 Performance

  • API Speed: REST queries completed under 200ms.
  • Relay: Critical warnings broadcasted to the doctor's screen in under 500ms.
  • Loads: Client assets compiled and rendered under 2.0 seconds.

3.5.2 Reliability

  • Integrity: Enforces database-level locks and atomic parameters during writes.
  • Continuity: Targeted continuous engine runtimes exceeding 30 days.

3.5.3 Availability

  • Uptime: Minimum continuous system uptime targeted at 99.5%.
  • Recovery: Frontend self-healing loops auto-retry dropped sockets every 5 seconds.

3.5.4 Security

  • Portal: Enforces strict stateless JWT token checks.
  • Encryption: Hashes user passwords using bcrypt (work factor 12) and secures all traffic using TLS 1.3.

3.5.5 Maintainability

  • Design: Enforces a modular Service Layer architecture, isolating business logic from controllers.
  • Layouts: UI structured using utility classes in Tailwind CSS v4 to keep components independent.

3.5.6 Portability

  • Browsers: Fully compatible across Chrome, Safari, Firefox, and Edge.
  • Environment: Deployable on standard hosts running PHP 8.2+ and Node 18+.

3.7 Design Constraints

  • Database: Exclusively limited to MongoDB Atlas document databases. SQL database structures are prohibited.

  • Data Retention: Telemetry collection entries archived to cold files after 12 months.

A.1 Appendix 1: Database Collection Schemas

The following tables define the structural schemas for our MongoDB collections.

1. Users Collection Schema

Field Name Data Type Key Type Validation / Rules
_id ObjectId Primary Key Unique, auto-generated
email String Index (Unique) Must be a valid email
password String Normal Hashed (bcrypt)
role String Index admin, doctor, or patient
is_verified Boolean Index Defaults to false

2. Health Metrics Collection Schema

Field Name Data Type Key Type Validation / Rules
_id ObjectId Primary Key Unique, auto-generated
patient_id ObjectId Index Foreign key reference to patients
heart_rate Number Normal Value must be between 30 and 220
spo2 Number Normal Value must be between 50 and 100

Method Endpoint Access Level Description
POST /api/v1/auth/register Public Registers a new user account
